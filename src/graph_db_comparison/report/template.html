Limitations

What This Benchmark Measures

  • Single-client query latency for a fixed set of openCypher operations (CREATE, MATCH, MERGE, shortestPath, aggregations, etc.)
  • Cold vs. warm execution time to reveal caching and JIT compilation effects
  • Feature coverage — which Cypher clauses, functions, and operators each database supports
  • Relative performance between databases on identical synthetic datasets at a single scale factor

Caveats

  • Synthetic data only. The generated social-graph dataset (persons, companies, KNOWS/WORKS_AT edges) does not reflect real-world data distributions, skew, or cardinality patterns.
  • Small scale. Default dataset sizes (1K–5K nodes, 5K–25K edges) are far below production graph sizes. Performance characteristics can change dramatically at millions of nodes/edges.
  • Single client, no concurrency. All benchmarks (except concurrent_writes) run from a single thread. This does not measure throughput under concurrent load.
  • No index tuning. Benchmarks use minimal indexing. Production deployments with proper indexes, constraints, and query hints will perform differently.
  • Network vs. embedded. Server-mode databases include network round-trip latency; embedded databases do not. The report tags each mode, but direct cross-mode comparisons are not apples-to-apples.
  • No durability or recovery testing. Write benchmarks do not measure fsync, WAL, replication, or crash-recovery behavior.
  • openCypher subset. Only features detectable by the opencypher-compliance package are tested. Proprietary extensions (e.g., Neo4j APOC, Memgraph MAGE) are not exercised.
  • Environment-dependent. Results depend on hardware, OS, Docker resource limits, and background load. They are not transferable across machines without re-running.

What This Benchmark Should Not Be Used For

  • Production workload sizing. These results cannot predict throughput, P99 latencies, or resource consumption for your specific data model and query mix.
  • High-concurrency or multi-tenant performance. The benchmarks do not simulate connection pooling, concurrent readers/writers, or tenant isolation.
  • Operational readiness. Backup/restore speed, monitoring integration, cluster failover, rolling upgrades, and operational maturity are not evaluated.
  • Cost analysis. Licensing costs, cloud instance pricing, and total cost of ownership are out of scope.
  • Query optimizer quality at scale. With small datasets, most queries complete via sequential scans. Optimizer differences only emerge with larger graphs and more complex access patterns.
  • Graph analytics or GDS workloads. PageRank, community detection, centrality, and other graph-algorithm workloads are not included.
